# تدفق الرسائل بعد إضافة الـ Cycle

هذا المستند يشرح الشكل الجديد لرفع ملف Excel الخاص بتوزيع المراقبين وكيفية ربطه بالإرسال.

## ما هي الـ Cycle

الـ `Cycle` هي جلسة امتحانية كاملة ناتجة عن عملية رفع واحدة.

كل مرة يتم رفع ملف Excel جديد:

- يتم إنشاء `Cycle` جديدة تلقائيًا
- يتم حفظ كل صفوف `EST1` و`EST2` تحت نفس الـ `Cycle`
- لا يتم تعديل أو استبدال أي بيانات قديمة
- تظل كل الدورات القديمة متاحة للمراجعة أو إعادة الإرسال أو المقارنة

بالتالي أصبح النظام يحتفظ بتاريخ الرفع شهريًا أو حسب كل جلسة امتحان بدل الاعتماد على جدول واحد يتم تكديس البيانات فيه بدون فصل.

## كيف يعمل رفع Excel الآن

ملف Excel يجب أن يحتوي على شيت واحدة أو اثنتين من النوع:

- `EST1`
- `EST2`

النظام يقرأ الشيتات المرئية ويقبل اختلافات الهيدر طالما المعنى معروف، مثل:

- `ROOM`
- `ROOM EST1`
- `ROOM EST2`
- `name`
- `Full English name (at least 4 names)`

ولا يهم:

- الحروف الكبيرة أو الصغيرة
- المسافات الزائدة
- الـ tabs
- اختلافات بسيطة في كتابة اسم العمود

بعد القراءة:

1. يتم توحيد الأعمدة إلى الحقول الداخلية للنظام.
2. يتم تحديد الشيت لكل صف (`EST1` أو `EST2`).
3. يتم إنشاء `Cycle` جديدة باسم مشتق من اسم الملف وتاريخ الرفع إذا لم يتم تمرير اسم صريح.
4. يتم إدخال الصفوف الصحيحة داخل قاعدة البيانات تحت نفس الـ `Cycle`.

## قواعد سلامة البيانات أثناء الاستيراد

الاستيراد لم يعد يتوقف بسبب صف واحد غير صالح.

الآن النظام:

- يتخطى الصفوف التي لا تحتوي على اسم
- يتخطى الصفوف التي لا تحتوي على بريد إلكتروني
- يتخطى الصفوف ذات البريد الإلكتروني غير الصحيح
- يتخطى الصفوف المكررة داخل نفس الرفع
- يكمل استيراد بقية الصفوف بشكل طبيعي

ويتم إرجاع عدد:

- `imported`
- `skipped`

مع أمثلة على الصفوف التي تم تخطيها وأسباب التخطي.

## كيف يتم تخزين البيانات

أصبح التخزين يعتمد على علاقتين أساسيتين:

- `RecipientCycle`
- `Recipient`

كل `Recipient` جديد يحتوي على:

- `cycleId`
- `sheet`

حيث:

- `cycleId` يربط المستلم بالجلسة الامتحانية التي جاء منها
- `sheet` يحدد هل الصف جاء من `EST1` أو `EST2`

أما البيانات القديمة فتبقى متاحة أيضًا، وتظهر كبيانات legacy بدون كسر التدفقات السابقة.

## كيف تظهر البيانات في الواجهة

في شاشة الرسائل أصبح هناك اختيار للدورة النشطة `Active cycle`.

يمكن للمستخدم:

- عرض أحدث دورة
- التبديل إلى أي دورة قديمة
- عرض كل الدورات معًا عند الحاجة

وعند اختيار دورة معيّنة:

- يتم تحميل المستلمين التابعين لها فقط
- يتم تحميل اللوجات الخاصة بها فقط
- يتم استخدام نفس الدورة في الإرسال

## الفلاتر الجديدة

الفلاتر النصية ما زالت متاحة مثل:

- الاسم
- البريد
- الغرفة
- العنوان
- المبنى
- الموقع

أما هذه الحقول فأصبحت `dropdown` ديناميكية بحسب بيانات الدورة الحالية:

- `Role`
- `Type`
- `Governorate`

بالتالي المستخدم لا يكتب قيمًا يدويًا وقديمة، بل يختار من القيم الموجودة فعليًا في ملف Excel الذي تم رفعه.

## كيف يعمل الإرسال الآن

الإرسال أصبح مرتبطًا بالدورة المختارة.

هذا يعني أن أوضاع الإرسال مثل:

- `Selected`
- `Filtered`
- `All pending`
- `Retry failed`

تعمل داخل نطاق الـ `Cycle` الحالية بدلًا من خلط كل بيانات الرفع القديمة مع الجديدة.

كذلك:

- حالة المستلم تتحول إلى `PROCESSING` عند بدء محاولة الإرسال
- ثم تتحول إلى `SENT` أو `FAILED` فور انتهاء المحاولة
- ويتم إنشاء `Log` لكل عملية إرسال

## التعامل مع Bulk Email

منطق الإرسال لم يعد sequential مستلمًا بمستلم، بل يعمل الآن على دفعات متوازنة `batched concurrency`.

هذا يحسن الإرسال في الحالات الكبيرة مثل آلاف الرسائل لأن:

- عدة مستلمين تتم معالجتهم في نفس الوقت
- حالة كل مستلم تتحدث فور انتهاء محاولته
- اللوجات تسجل نتيجة كل صف بشكل مستقل

الإعداد الافتراضي الحالي يستخدم توازيًا آمنًا ويمكن التحكم فيه عبر:

- `MESSAGING_SEND_CONCURRENCY`

ملاحظة مهمة:

إذا كانت البنية المستضيفة نفسها تفرض request timeout قصير جدًا، فالإرسال لعدد ضخم جدًا مثل `10000` مستلم قد يحتاج مستقبلاً worker أو queue خارجية. لكن داخل البنية الحالية تم تحسين المعالجة بشكل واضح، وأصبحت أفضل كثيرًا من التنفيذ المتسلسل القديم.

## ماذا تم التحقق منه عمليًا

تم تنفيذ اختبار فعلي end-to-end على الملف:

- `updated_emails_cleaned.xlsx`

والنتائج كانت كالتالي:

- إنشاء `Cycle` جديدة بنجاح
- استيراد `1568` صفًا صحيحًا
- تخطي `35` صفًا غير صالح
- توزيع البيانات بين:
  - `EST1 = 1004`
  - `EST2 = 564`
- تحميل قوائم `Role` و`Type` و`Governorate` من نفس البيانات المستوردة
- إرسال بريد إلكتروني حقيقي بنجاح إلى مستلمين داخل الدورة
- تحديث الحالات إلى `SENT`
- تسجيل اللوجات بنجاح

كما تم اختبار edge cases إضافية:

- صف مكرر
- صف بدون بريد
- صف ببريد غير صالح
- صف بدون اسم

وتم تخطيها بدون تعطيل الاستيراد.

## الخلاصة

النظام الآن يعمل بهذا المفهوم:

`Excel Upload -> Create Cycle -> Import EST1/EST2 -> Filter by Cycle -> Send داخل نفس الدورة -> Log per recipient`

وهذا يجعل النظام:

- قابلًا للتتبع
- مناسبًا للرفع الشهري أو لكل جلسة امتحان
- أكثر أمانًا في البيانات
- أوضح في التشغيل والإرسال والمراجعة
